# Virtual Bond / 虚拟纽带

AI 驱动的桌面陪伴应用。项目希望建立一种平等、持续、可沉淀记忆的数字伙伴关系，而不只是一个会弹出对话框的桌宠。

## 项目状态

当前版本为 `0.1.0`，处于桌面原型阶段。桌面应用壳、双模式窗口、本地对话持久化和真实流式 AI 对话已经可用，角色动画和长期记忆仍在开发计划中。

### 已实现

- Electron 透明无边框桌面窗口
- 可拖拽、置顶并记忆位置的悬浮桌宠，支持 Windows 高 DPI 稳定拖动与透明空白区域鼠标穿透
- 可在 `45%–240%` 范围调整桌宠显示比例，窗口与交互区域同步缩放，操作按钮保持易用尺寸并避开人物脚部
- 桌宠与完整陪伴空间双模式切换
- 关闭陪伴空间时自动返回桌宠，桌宠状态再负责隐藏或退出应用
- 桌宠缩放使用固定逻辑画布的渲染层变换，不再改变 Electron 页面缩放；`45%–240%` 下往返陪伴空间不会残留视口偏移或房间画面
- 系统托盘、右键菜单和 `Ctrl+Shift+B` 显示/隐藏快捷键
- 单实例运行和关闭后驻留托盘
- Vue 3 + TypeScript 陪伴空间界面
- SQLite 角色、对话和应用设置持久化
- 默认角色“牧濑红莉栖”和真实流式对话，本地验收默认使用 DeepSeek 直连
- OpenAI Responses、Claude、Gemini 和自定义 OpenAI 兼容 API
- Electron `safeStorage` 加密密钥与模型设置界面
- CCSwitch 当前配置、分享链接和复制 JSON 导入
- 停止生成、超时以及常见 API 错误提示
- 基于 context isolation、preload 和 IPC 的受控桌面 API
- 数据驱动皮肤清单解析、用户皮肤目录扫描和路径安全校验
- 皮肤预览、选择持久化、手动重新扫描、用户目录入口和损坏资源错误提示
- PixiJS 规则网格 Sprite Sheet 播放、PNG 尺寸校验和 CSS 桌宠降级
- PixiJS 严格 CSP 兼容以及桌宠到陪伴空间的同步模式切换
- PixiJS 场景与皮肤播放器按 ticker、观察器、显示对象、renderer 的顺序安全卸载，避免模式切换时残留画面或访问已销毁 geometry
- 实体皮肤加载期间保持透明，只有确认资源不可用时才显示 CSS Q 版降级形象；设置页预览同样区分加载中与不可用状态
- 桌宠和陪伴空间复用同一套皮肤与待机动画
- 陪伴空间使用全窗口布局，并根据舞台尺寸动态计算角色大小与脚底锚点
- 陪伴空间为 Q 版 PixiJS 模拟经营式房间场景：`960 × 640` 逻辑坐标、脚底 Y 深度排序、家具前后分层、雨夜公寓配色，程序化绘制窗景、书架、书桌、餐桌、阅读椅和植物
- 房间行为由独立的 TypeScript 动作状态机驱动（idle / walk / sit / look-around / thinking / speaking / studying / eating / resting / interaction），带优先级、锁定、打断与回退，并与 PixiJS 渲染分离、可单元测试
- 陪伴空间提供自由陪伴 / 一起学习 / 一起吃饭 / 休息一下四种情境入口：切换后角色走向对应家具并进入活动，选择通过 localStorage 持久化；点击地毯或家具可即时互动，对话状态会驱动思考与说话表现，空闲时低频触发看窗等环境动作并遵循减少动态效果偏好
- 多层剖面陪伴空间原型：三层八个可点击空间、中央升降井、情境到房间映射以及可测试的跨楼层节点路径；跨层路线强制先水平到达电梯，经过楼层节点离散跳转后再前往房间，避免斜线穿层
- 牧濑红莉栖 Q 版角色设定板与首套 8 帧向右行走原型已接入场景；生成后的 Sheet 会自动去除色键并统一每帧人物高度、水平中心和脚底基线
- 桌宠动作选择已统一为声明式状态：拖拽、气泡说话、点击回应和待机按优先级切换；皮肤缺少对应素材时稳定回退到 `idle`
- 陪伴空间按当前显示器工作区收敛窗口，长聊天只在消息区滚动，并同步 PixiJS 首帧画布尺寸，保证首次进入时人物全身、快捷操作和设置入口完整可见
- 每帧 1024 像素的参考图测试皮肤、下半身锁定呼吸动画和 4K 角色母版
- 透明边缘去色污染、线性 mipmap 采样和自适应 3×–4× 的 PixiJS 超采样渲染
- 从 4K 母版本地重导出的高清待机帧，以及约 4 秒一周期、胸肩主导并带最小可见头颈联动的呼吸
- 桌宠气泡只在角色实际说话时出现，并按文本长度自动延时淡出
- 面向图片模型的中文 Sprite Sheet 生成 Prompt 模板
- 与普通聊天隔离的 GPT Image 2 图片 Provider 与安全密钥
- Windows、macOS 和 Linux 打包配置

## 当前架构

```text
src/
├─ main/
│  ├─ index.ts       # Electron 窗口、托盘、快捷键和桌宠拖拽
│  ├─ database.ts    # Electron 数据库入口
│  ├─ database-core.ts     # SQLite 迁移、角色、对话和设置
│  ├─ chat-provider.ts      # 多 Provider 流式模型调用
│  ├─ provider-settings.ts  # 加密密钥、设置和 CCSwitch 导入
│  ├─ image-provider.ts     # 独立图片 Provider 与安全设置
│  ├─ window-layout.ts      # 房间窗口边界与最小尺寸计算
│  └─ ipc.ts                # 主进程业务 IPC
├─ preload/
│  ├─ index.ts       # 暴露给渲染进程的受控 API
│  └─ index.d.ts     # API 类型声明
└─ renderer/
   └─ src/
      ├─ App.vue     # 陪伴空间和对话界面
      ├─ PetView.vue # 悬浮桌宠界面
      ├─ PetSpritePlayer.vue # PixiJS 通用序列帧播放器
      └─ SettingsPanel.vue # 模型设置界面
```

主进程拥有窗口、数据库、密钥和模型调用能力；渲染进程只通过 preload 暴露的 API 调用这些能力。用户消息和完成或主动停止后的伙伴回复会写入 SQLite。

## 开发

建议使用 Node.js LTS。

在另一台电脑接续开发时，先阅读 [`docs/development-status.md`](docs/development-status.md)。该文件随每次推送更新，记录最新提交、验证结果、本机状态边界和下一步入口。

涉及窗口和角色布局的改动，实机验收至少覆盖普通窗口、最大化、连续拖动调整尺寸以及返回桌宠四种状态。

```bash
npm ci
npm run dev
```

质量检查：

```bash
npm run typecheck
npm test
npm run lint
npm run build
```

一次运行全部检查：

```bash
npm run check
```

打包：

```bash
npm run build:win
npm run build:mac
npm run build:linux
```

如果 Electron 二进制下载超时，可以临时配置镜像后重新安装：

```powershell
$env:ELECTRON_MIRROR = 'https://npmmirror.com/mirrors/electron/'
npm ci
```

完整技术选型见 [`docs/tech-stack.md`](docs/tech-stack.md)。

皮肤包、桌宠缩放和动画资源规范见 [`docs/skin-system.md`](docs/skin-system.md)，中文 Sprite Sheet 生图模板见 [`resources/prompts/sprite-sheet.zh-CN.md`](resources/prompts/sprite-sheet.zh-CN.md)。

Q 版角色、移动状态、跨楼层动作和生成后处理方案见 [`docs/character-animation.md`](docs/character-animation.md)。

图片 API、安全密钥、标准角色参考、1K 模板到 4K 优化流程和高质量测试皮肤见 [`docs/image-generation.md`](docs/image-generation.md)。

本轮开发中遇到的环境、Electron、SQLite、CCSwitch、流式 IPC、图片生成、PixiJS 和动画素材问题，见 [`docs/troubleshooting.md`](docs/troubleshooting.md)。

## 开发计划

计划按“先完成可用对话闭环，再增加表现和长期关系能力”的顺序推进。

### `0.2.0`：真实 AI 对话闭环

目标：让用户配置自己的 API Key，并与牧濑红莉栖进行真实、流式、可恢复的模型对话。

- [x] 定义统一的流式 Provider 调用层
- [x] 支持 OpenAI Responses、Claude、Gemini 和自定义 OpenAI 兼容 API
- [x] 使用 Electron `safeStorage` 加密保存 API Key
- [x] 增加模型、API 地址和角色附加指令设置
- [x] 支持流式回复、停止生成和超时
- [x] 组装最近对话和角色状态作为模型上下文
- [x] 增加有限自动重试和可持久化的消息状态

验收标准：

- 用户不需要修改源码即可配置模型
- 发送消息后可以实时看到流式回复
- 应用重启后能继续已有对话
- Key 缺失、网络失败和接口限流都有明确提示
- API Key 不以明文写入数据库或日志

### `0.3.0`：数据可靠性与工程基础

目标：为后续记忆、角色和玩法功能建立可升级、可测试的数据层。

- [x] 实现真正的数据库版本迁移
- [x] 增加 IPC 参数校验和消息长度限制
- [x] 将消息写入、回复状态更新设计为可恢复流程
- [x] 拆分 Provider、数据库、对话和设置模块
- [x] 增加数据库迁移与消息状态单元测试
- [x] 增加 GitHub Actions
- [x] 统一 Git 换行符与 Prettier 配置
- [ ] 增加 Provider 单元测试和关键 IPC 集成测试

验收标准：

- 旧版本数据库可以无损升级
- AI 请求中断后不会留下无法处理的半成品数据
- 核心数据层和 IPC 有自动化测试保护
- Typecheck、Lint、测试和构建可在 CI 中通过

### `0.4.0`：角色视觉与状态

目标：用真实角色素材替换 CSS 占位角色，让桌宠状态能响应对话和系统事件。

- [x] 增加可持久化的桌宠显示比例，并同步缩放窗口、点击区域和气泡
- [x] 建立 PixiJS 角色渲染组件
- [x] 定义开放式皮肤目录与 `manifest.json`，仅将 `idle` 设为必选动作
- [x] 动态加载规则网格 sprite sheet，不在加载器中硬编码动作名称和帧数
- [x] 支持动作 FPS、循环方式、脚底锚点及动作缺失回退
- [x] 增加皮肤预览、选择持久化、手动重新扫描和错误提示界面
- [x] 建立可打断、可回退的通用动作状态机
- [x] 将陪伴空间从全身立绘展示改为 Q 版房间形态，使用 PixiJS 场景图、脚底深度排序、可行走区域和家具互动锚点
- [ ] 桌宠模式支持点击区域和自动水平翻转
- [ ] 增加文件变化监听与开发模式热重载
- [ ] 为行走等移动动作增加屏幕安全区、活动半径与低打扰策略

验收标准：

- 桌宠和陪伴空间使用同一套角色状态模型
- 更换符合规范的皮肤或增加普通动画时无需修改加载器代码
- 调整尺寸、重启应用及切换显示器后，桌宠尺寸与位置保持正确
- 动画切换不会导致窗口跳动或明显性能下降
- 素材损坏或缺失时应用仍可进入对话

### `0.4.5`：日常情境与房间行为

目标：让陪伴空间的房间、位置和动作能够对应用户明确表达的日常状态，而不是只播放随机动画。

- [x] 桌宠模式继续使用高清全身形态，陪伴空间使用同一角色的 Q 版房间形态
- [x] 先支持用户主动选择学习、用餐、休息和自由陪伴模式（可编辑日程留待后续）
- [x] 将用户情境统一转换为伙伴意图，再选择房间、家具锚点和动作
- [x] 学习时可在书桌执行阅读或查资料表现，用餐时移动到餐桌，休息时选择座椅或窗边
- [x] 为情境来源、优先级、中断和回退提供可解释状态，不直接监控用户前台应用
- [ ] 接入可编辑的时间日程
- [ ] Agent 接入后再把“查资料”等表现动作升级为真实搜索、整理和任务反馈

验收标准：

- 用户可以明确控制当前情境，并知道角色为何进入某个房间或动作
- 房间切换、行走和家具互动共享同一套伙伴意图与动作状态机
- 关闭日程联动后不会根据时间或系统活动自行推断用户状态
- 没有 Agent 或网络不可用时仍可降级为本地陪伴表现

### `0.5.0`：多层剖面陪伴空间

目标：让角色生活在可观察、可移动、有环境叙事的多层建筑中，而不是停留在单层 Q 版房间和固定控制面板里。

- [x] 建立三层八房间的程序化剖面原型
- [x] 建立情境到房间映射和跨楼层节点路径

- [ ] 完成强角色轮廓与多层空间概念图
- [ ] 增加镜头跟随、楼层聚焦和房间内物件交互
- [ ] 建立动作精灵图规格并制作第一批运行素材
- [ ] 把设置、日程与长期记忆入口映射为空间内物件

验收标准：

- 至少三层、六个可到达房间
- 四种情境分别映射到不同空间
- 角色可以跨房间和跨楼层移动

- 普通窗口、最大化和连续调整尺寸时保持完整剖面比例

## 已知工程问题

- 历史 Windows 检出文件可能仍保留 CRLF；新检出由 `.gitattributes` 统一为 LF
- 项目路径包含空格时，`better-sqlite3` 原生重建可能产生 `node-gyp` 警告
- 当前完整陪伴空间可能继续保持始终置顶，需要进一步确认交互预期
- 当前人物点击与拖拽范围使用皮肤 `hitbox` 的矩形包围盒，逐像素 Alpha 命中仍待实现
- Provider 和 IPC 自动化覆盖仍不完整，模型调用需要按实际服务手工回归
